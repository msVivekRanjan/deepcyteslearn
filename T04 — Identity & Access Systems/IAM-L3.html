Module 1
🎫 Ticketing Systems
Understanding access request workflows, approvals, and audit trails in enterprise environments.

Intro What is a Ticketing System?

▾

A ticketing system is a centralized platform that manages, tracks, and resolves access requests, incidents, and service requests in an organization. In IAM, ticketing systems are the backbone of the access request lifecycle.

Popular platforms include ServiceNow, Jira Service Management, and Remedy. These tools enforce governance by ensuring every access change is documented, approved, and traceable.

Key concepts:

  • Access Request: Formal request to gain access to a system or resource
  • Approval Chain: Defined hierarchy of approvers (Manager → IT → Security)
  • Audit Trail: Complete history of who approved what, and when
  • SLA: Service Level Agreement — time limits for request resolution
  • Compliance Tracking: Evidence for SOX, PCI-DSS, ISO 27001 audits

Example 🏦 Real World: Bank SAP Access Request

▾

Raj Kumar, a new Finance Analyst at State Bank, needs access to the SAP Finance module to process payments.

The workflow:

  • Raj raises a ticket in ServiceNow requesting SAP Finance Read access
  • His manager Priya Sharma receives an email notification and approves
  • The request escalates to the IT Security team for risk validation
  • IT provisions the account in SAP and closes the ticket
  • Raj receives access confirmation. The ticket is archived for compliance evidence

Without this system, access could be granted informally via email — creating security risks, audit failures, and compliance gaps.

🛡️ Conditional Access
Context-aware, risk-based authentication policies using Zero Trust principles.

Theory What is Conditional Access?

▾

Conditional Access is a security approach where access decisions are based on real-time context — not just username and password. It evaluates signals to decide whether to allow, require MFA, or block access entirely.

Key signals evaluated:

  • User identity: Who is trying to access? (Employee, Contractor, Admin)
  • Device compliance: Is the device managed and patched?
  • Location: Is the user in a trusted location or an anomalous country?
  • Risk score: Has the account been flagged for suspicious behavior?
  • Application sensitivity: Is this a low-risk email or high-risk financial system?

Zero Trust principle: "Never trust, always verify." Even if a user is inside the corporate network, they must prove their identity and context for every access request.

Module 4
🔑 Privileged Access Management
Securing, controlling, and monitoring privileged accounts with vaults, JIT access, and session recording.

Theory What is PAM?

▾

Privileged Access Management (PAM) is a security discipline focused on protecting accounts with elevated access to critical systems. Privileged accounts — like Domain Admins, Root accounts, or Database Admins — can cause catastrophic damage if compromised.

Core PAM capabilities:

  • Password Vault: Stores privileged credentials encrypted. Admins never see the real password.
  • Credential Checkout: Admins "check out" a credential for a specific time window
  • Session Recording: Every keypress and screen action is recorded during privileged sessions
  • Just-In-Time (JIT) Access: Access is granted only when needed, then automatically revoked
  • Credential Rotation: Passwords are automatically changed after each use

Enterprise PAM tools: CyberArk, BeyondTrust, Thycotic, Delinea

Module 5
🔗 Identity Federation & SSO
SAML, OAuth 2.0, OpenID Connect — enabling seamless single sign-on across enterprise applications.

Theory Federation Protocols

▾

Identity Federation allows users to authenticate once and access multiple systems — even across organizational boundaries. This eliminates password fatigue and centralizes identity governance.

Key protocols:

  • SAML 2.0: XML-based standard for SSO between Identity Provider (IdP) and Service Provider (SP). Used in enterprise apps like Salesforce, SAP.
  • OAuth 2.0: Authorization framework for delegated access. Allows apps to access resources on behalf of users without sharing passwords.
  • OpenID Connect (OIDC): Authentication layer on top of OAuth 2.0. Returns an ID Token (JWT) with user identity claims.

Key terms: Identity Provider (IdP) — authenticates the user. Service Provider (SP) — relies on IdP for authentication. Claims — attributes about the user (email, role, department). Token — cryptographically signed proof of authentication.

Module 6
🔄 Identity Lifecycle Management
Joiner, Mover, Leaver — managing identities from onboarding to offboarding.

Theory The JML Framework

▾

Identity Lifecycle Management governs how digital identities are created, maintained, and removed throughout an employee's journey.

  • 🟢 Joiner: New employee joins. Account created, roles assigned based on job function. Access provisioned on Day 1.
  • 🟡 Mover: Employee transfers to another department. Access to old department revoked; new department access provisioned. Role-mapping reviewed.
  • 🔴 Leaver: Employee exits the organization. All accounts immediately disabled, tokens revoked, group memberships removed. Audited for compliance.

Failure to execute the Leaver process creates orphan accounts — active accounts with no known owner — a top security risk in organizations.

Module 10
👁️ Identity Monitoring
Real-time threat detection, anomaly analysis, and SIEM integration for IAM events.

Theory Identity Threat Detection

▾

Identity monitoring continuously analyzes authentication events, privileged activity, and access patterns to detect threats in real time.

  • Failed Login Monitoring: Brute force attacks, credential stuffing
  • Impossible Travel: Login from Mumbai at 9AM, then London at 9:05AM — physically impossible
  • Privileged Account Abuse: Admin accessing systems outside business hours
  • Lateral Movement: Account accessing systems it has never accessed before
  • SIEM Integration: Forward all events to Splunk, Microsoft Sentinel, IBM QRadar for correlation

Module 2
🤝 Delegation
Administrative delegation, temporary authority, and separation of duties in IAM workflows.

Theory Delegation Concepts

▾

Delegation in IAM allows authorized users to temporarily transfer their administrative rights or approval authority to another trusted individual. This is critical for business continuity — especially when managers are on leave.

  • Administrative Delegation: Delegating IT admin rights (e.g., creating users in AD)
  • Approval Delegation: Delegating access request approvals to a deputy
  • Temporary Authority: Time-bounded — automatically revokes after the set period
  • Separation of Duties (SoD): Ensures no single person has end-to-end control of a sensitive process

Module 7
⚖️ Access Governance
Access reviews, certification campaigns, SoD enforcement, and least privilege management.

Theory Access Governance Fundamentals

▾
  • Access Reviews: Periodic review of all user entitlements — managers certify "keep" or "remove"
  • Certification Campaigns: Scheduled reviews across departments (quarterly, annual)
  • Segregation of Duties (SoD): Preventing conflicts like "Create PO + Approve PO" by same person
  • Least Privilege: Users have only the minimum access required for their job — nothing more

Module 8
👑 Privileged Accounts
Identifying, securing, and governing high-risk privileged identities across the enterprise.

Theory Types of Privileged Accounts

▾
  • Domain Admin: Full control of Active Directory. Can create users, GPOs, reset any password.
  • Root Account: Superuser on Unix/Linux systems. No access restrictions.
  • Cloud Admin: Global Administrator in Azure/AWS. Can access all cloud resources.
  • Database Admin (DBA): Full access to all databases — production data, PII, financial records.
  • Service Accounts: Non-human accounts running automated processes. Often have high privileges and are overlooked.

Module 9
🔗 Authentication Chains
Multi-factor authentication flows from username to biometrics.

Theory Authentication Factors

▾
  • Something You Know: Password, PIN, security question
  • Something You Have: OTP token, authenticator app, hardware key (YubiKey)
  • Something You Are: Fingerprint, Face ID, retina scan
  • Context: Location, device, time-of-day (adaptive factor)

MFA combines two or more factors. A compromised password alone cannot bypass OTP or biometrics.
