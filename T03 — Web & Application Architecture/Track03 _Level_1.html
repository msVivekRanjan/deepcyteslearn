01
Browsers
The software that fetches and displays web pages for you
📖 THEORY

A web browser is an application that retrieves, processes, and renders content from the internet. When you type a URL and press Enter, the browser performs a sequence of steps: it looks up the server's IP address via DNS (Domain Name System), opens a TCP connection, sends an HTTP request, receives the server's response, and finally renders the page on your screen.

🔧 Rendering Engine
The browser's core component that parses HTML & CSS and paints pixels. Chrome/Edge use Blink, Firefox uses Gecko, Safari uses WebKit.
⚡ JavaScript Engine
Executes JS code. Chrome uses V8 which compiles JS to machine code (JIT), making it extremely fast. Firefox uses SpiderMonkey.
🌳 The DOM
After parsing HTML, the browser creates the Document Object Model — a tree structure of all elements. JavaScript reads and modifies this tree to make pages interactive.
🗄 Browser Storage
Browsers store data locally using Cookies, localStorage, sessionStorage, and IndexedDB for offline capability and performance.
🏢
Analogy: Think of a browser as a post office + translator. It knows the address (URL), retrieves your mail (HTML), and then translates the coded instructions into a beautiful, readable letter (the webpage you see).

02
Web Servers
Computers that listen for requests and send back responses
📖 THEORY

A web server is software (running on a physical or virtual computer) that listens for incoming HTTP requests on a specific port — usually port 80 for HTTP and port 443 for HTTPS — and sends back a response. The response could be a static file (HTML, image, CSS) or dynamically generated content from a database.

📂 Static vs Dynamic
A static server serves files directly from disk (like a CDN serving images). A dynamic server runs code, queries databases, and builds the response on the fly.
🔌 Ports
Ports are like doors on a computer. Port 80 = HTTP, Port 443 = HTTPS, Port 3000/8080 = common dev servers. A computer can have many servers running on different ports simultaneously.
⚖️ Load Balancing
High-traffic sites run multiple server instances. A load balancer distributes incoming requests across them so no single server gets overwhelmed.
🔁 Reverse Proxy
Nginx is often used as a reverse proxy — it sits in front of your app server, handles SSL termination, caches responses, and forwards requests to the right backend service.
🏨
Analogy: A web server is like a hotel front desk. Guests (browsers) arrive and make requests ("I want room 404"). The receptionist (server) finds and delivers what was asked, or politely says "sorry, that room doesn't exist."

03
URLs
Uniform Resource Locators — the address of every web resource
📖 THEORY

A URL (Uniform Resource Locator) is a standardized address that uniquely identifies a resource on the internet. Every image, webpage, video, and API endpoint has its own URL. A URL is made up of several distinct parts, each serving a specific purpose in locating the exact resource you want.

🔍 DNS Resolution
The hostname in a URL (e.g. google.com) gets converted to an IP address (e.g. 142.250.80.46) by DNS servers — the internet's phone book. Without DNS, you'd need to memorize IP addresses.
📦 URL Encoding
Special characters in URLs must be percent-encoded. A space becomes %20, & becomes %26. This ensures URLs can only contain safe ASCII characters.
🌐 Absolute vs Relative
Absolute URLs contain the full address (https://site.com/page). Relative URLs are shortcuts like /about or ../images/logo.png, interpreted relative to the current page.
🔑 Query vs Path Params
/users/42 — path parameter (part of the resource identity). ?sort=asc — query parameter (filters/options). APIs use both: path for "what", query for "how".

04
HTTP
HyperText Transfer Protocol — the language of the web
📖 THEORY

HTTP (HyperText Transfer Protocol) is the foundation of data communication on the web. It's a request-response protocol — a client (browser) sends a request, and a server sends back a response. HTTP is stateless, meaning each request is completely independent; the server has no memory of previous requests from the same client.

📨 HTTP Request Structure
Every request has: a Method (GET/POST…), a URL (target resource), Headers (metadata like Accept, Authorization), and optionally a Body (data for POST/PUT).
📬 HTTP Response Structure
Every response has: a Status Code (200, 404…), Headers (Content-Type, Set-Cookie…), and a Body (the HTML, JSON, or file content).
🚀 HTTP/1.1 vs HTTP/2 vs HTTP/3
HTTP/1.1 — one request at a time per connection. HTTP/2 — multiplexing (many requests at once). HTTP/3 — uses QUIC protocol over UDP for even faster connections.
📋 Key Headers
Content-Type tells what format the body is in. Authorization carries auth tokens. Cache-Control tells browsers how long to cache the response.
📮
Analogy: HTTP is like sending a letter. You write what you want (request), put it in an envelope with an address (URL), and the post office (internet) delivers it. The recipient reads it and sends a reply (response). Each letter is independent — they don't remember past letters.

05
HTTPS
HTTP + TLS/SSL encryption — making it safe
📖 THEORY

HTTPS (HTTP Secure) is HTTP with an added layer of encryption provided by TLS (Transport Layer Security) — formerly known as SSL. Before any HTTP data is sent, TLS performs a handshake that authenticates the server using a digital certificate and establishes a shared encryption key. All data is then encrypted in both directions.

🔐 What TLS Provides
Confidentiality — data is encrypted, can't be read by eavesdroppers. Integrity — data can't be tampered with. Authentication — proves you're talking to the real server, not an imposter.
📜 SSL Certificates
A Certificate Authority (CA) like Let's Encrypt or DigiCert digitally signs a server's certificate, vouching for its identity. Browsers have a built-in list of trusted CAs and reject unknown ones.
🔑 Asymmetric + Symmetric
The handshake uses asymmetric encryption (public/private keys) to safely exchange a session key. Actual data transfer uses faster symmetric encryption (AES) with that shared key.
⚡ Performance
Modern TLS 1.3 adds only ~1ms of overhead for the handshake. HTTPS is effectively free in terms of performance and is now required for HTTP/2, which is actually faster than HTTP/1.1.
🔒
Analogy: HTTPS is like sending documents in a locked briefcase. Before you start, you and the recipient agree on a lock combination using a secure method no one else can overhear. After that, everything travels in locked cases — even if someone intercepts it, they can't open it.

06
Cookies
Small pieces of data stored in your browser by websites
📖 THEORY

A cookie is a small key-value pair (up to 4KB) that a server sends to the browser via a Set-Cookie header. The browser stores it and automatically attaches it to every subsequent request to the same domain. Cookies solve the problem of HTTP being stateless — they give the server a way to recognise you across requests.

🍪 Session Cookies
Cookies without an Expires or Max-Age attribute are session cookies — they're deleted when you close the browser tab. Used for temporary login states.
📅 Persistent Cookies
Cookies with an Expires date persist even after the browser is closed. Used for "remember me" features, preferences, and long-lived authentication tokens.
🚫 Third-Party Cookies
Set by a different domain than the one you're visiting (e.g. ad networks). Used for cross-site tracking. Modern browsers are phasing these out for privacy reasons.
⚠️ Security Risks
XSS attacks can steal cookies if JS can read them — use HttpOnly to prevent this. CSRF attacks forge requests using your cookies — use SameSite=Strict to block them.
🎫
Analogy: Cookies are like a loyalty card stamp from a café. The café (server) gives you a stamp (cookie) on your first visit. Every time you return, you show the stamp and they know you're a regular customer — without needing you to introduce yourself each time.

07
Sessions
Tracking a user across multiple requests — remembering who you are
📖 THEORY

A session is a server-side mechanism for maintaining state across multiple HTTP requests from the same user. When you log in, the server creates a session record in its memory or database and gives you a unique Session ID (a long random string) stored in a cookie. On each subsequent request, your browser sends this ID, and the server looks up your session to know who you are.

🗄 Session Storage
Sessions can be stored in server memory (fast, but lost on restart), a database (persistent), or a Redis cache (fast + persistent, used by large apps with multiple servers).
🆚 JWT Tokens
Modern apps often use JWT (JSON Web Tokens) instead of sessions. The server signs a token with user data. The client stores and sends it. No server-side storage needed — the token is self-contained.
⏱ Session Expiry
Sessions should expire after a period of inactivity (typically 30 mins–2 hours) or after a maximum duration for security. Banking apps expire sessions very quickly; social media keeps them longer.
🔐 Session Fixation
A security attack where an attacker forces a victim to use a known session ID. Prevention: regenerate the session ID after login so the pre-login ID becomes invalid.
🎪
Analogy: A session is like a theme park wristband. When you enter (login), they give you a wristband (session ID). For each ride (request), you show your wristband. They check it against their records to know your age, pass type, and access level — without asking you to prove identity each time.

08
APIs
Application Programming Interfaces — how apps talk to each other
📖 THEORY

An API (Application Programming Interface) is a contract that defines how two pieces of software should communicate. On the web, we mostly use REST APIs — they use HTTP methods and URLs to represent operations on resources. When your phone app shows the weather, it's calling a weather API. When you pay with Stripe, your app calls the Stripe API.

📐 REST Principles
REST (Representational State Transfer) has key rules: use HTTP methods correctly, resources are identified by URLs, responses are stateless, use JSON for data exchange, and use standard status codes.
🔐 API Authentication
APIs are usually protected. Common methods: API Keys (a secret string in headers), OAuth 2.0 (for user permissions), and JWT tokens (signed tokens with user identity).
📊 REST vs GraphQL
REST has fixed endpoints (GET /users/1). GraphQL has one endpoint and lets clients request exactly the fields they need — no over-fetching or under-fetching. Used by Facebook, GitHub.
⚡ Rate Limiting
APIs limit how many requests you can make per time window (e.g. 100 req/minute) to prevent abuse. Exceeding the limit returns a 429 Too Many Requests status code.
🍽️
Analogy: An API is like a restaurant menu. The menu (API docs) tells you what you can order (endpoints). You tell the waiter (API) your order. The kitchen (server) prepares it. You get your food (data) — without ever entering the kitchen or knowing how it was made.

Databases
Where structured information is organized and permanently kept
📖 THEORY

A database is an organized system used by application backends to store, access, modify, and manage information permanently. While application code variables are wiped whenever a server restarts, databases run as independent services to ensure data durability and integrity.

📊 Relational (SQL)
Organizes records into strict tables with predefined columns. Employs Primary Keys to identify items and Foreign Keys to build links between tables. Uses PostgreSQL, MySQL, SQL Server.
📦 Non-Relational (NoSQL)
Maintains flexible storage formats without predefined schemas. Often saves elements as JSON-like documents or key-value entries. Excellent for changing data maps. Uses MongoDB, Redis.
⚡ Indexing
A background data map created on specific table columns. An index stops slow row-by-row lookups, turning massive search tasks into super-fast lookups.
🛡️ ACID Transactions
A set of properties ensuring reliable operations: Atomicity (all or nothing), Consistency, Isolation, and Durability. Essential for things like banking balances.
🗄️
Analogy: A relational database is like a collection of highly organized spreadsheets linked together. If you update a customer's address in the main Customer list, every Order spreadsheet referencing that customer immediately reflects the new location.

Frontend vs Backend
Splitting the view you interact with from the hidden logical machine
📖 THEORY

Modern web applications split code responsibilities into two separate layers: the Frontend (Client-Side), which runs right inside the visitor's web browser, and the Backend (Server-Side), which executes remotely on dedicated infrastructure away from the user's view.

🎨 Frontend Tasks
Constructs layout structure, controls element styles, tracks button gestures, and updates display views instantly. Built using languages and runtimes like HTML, CSS, JavaScript, React, Vue.
⚙️ Backend Tasks
Maintains core application business logic, processes secure payments, updates database fields, and verifies identity rules. Powered by servers like Node.js, Python, Java, PHP, Ruby.
📡 The Connection API
The frontend doesn't talk directly to database tables for security reasons. Instead, it sends secure HTTP network messages to backend routes to modify data.
🔒 Security Context
Client code can always be examined using browser inspection menus. Private secret keys, proprietary formulas, or credential strings must always remain protected on the backend server.
🎭
Analogy: Think of a web app like a theater production. The frontend is the stage crew, visual props, and actors you see in front of you. The backend represents the backstage managers, lighting switches, and writers orchestrating the narrative out of sight.
