✓ Regenerate after login Prevents session fixation
✓ Logout invalidation Destroy server-side token

Access Control Models

RBAC Permissions tied to roles. Simple and scalable for most apps.
ABAC Access based on attributes: department, location, time of day.
DAC Resource owners grant permissions. Used in file systems.
MAC Central authority controls all permissions. Used in government/military.
Topic 07 · Credentials

🔑 Secrets Management

Secrets (API keys, passwords, tokens, certificates) must never be hardcoded in source code. Attackers regularly scan public repositories for exposed credentials.

Bad vs Good — Spot the Difference

Hardcoded secret (BAD) vs. Vault-managed secret (GOOD)
❌ app.js (committed to GitHub)
// Line 42
const DB_PASS = "P@ssw0rd!123";
const API_KEY = "sk-live-abc123xyz456";
// EXPOSED to anyone with repo access!
✅ app.js (safe approach)
// Load from environment / vault
const DB_PASS = process.env.DB_PASSWORD;
const API_KEY = await vault.getSecret('api-key');
// Secrets never in source code!

Secret Rotation Simulator

Rotate secrets regularly to limit exposure windows
sk-live-abc123xyz456...<EXPOSED IN REPO>

Best Practices

✓ Never hardcode secrets in code or config files
✓ Encrypt secrets at rest using a key management system
✓ Rotate credentials on a schedule and after any suspected breach
✓ Audit access — log every time a secret is retrieved
✓ Use tools like HashiCorp Vault, AWS Secrets Manager, or Azure Key Vault

Topic 09 · Monitoring

📋 Logging

Logs are your security camera system. Without them, you can't detect attacks, investigate incidents, or prove compliance. But logs must never contain sensitive data like passwords.

Live Log Stream

Simulated security event log

What to Log (and What Not To)

✅ DO LOG
• Login attempts (success/fail)
• Permission changes
• API request IDs and status
• Security events
• System errors
❌ NEVER LOG
• Passwords (even hashed)
• Full credit card numbers
• Session tokens
• Private keys
• PII beyond what's needed

Topic 11 · Process

🔄 Secure SDLC

Secure SDLC integrates security at every phase of software development. Fixing a vulnerability during design costs 10× less than fixing it after deployment — and 100× less than fixing it after a breach.

Phase Explorer — Click to Expand

Click each phase to see security activities
●
📋 1. Requirements
Identify security requirements and compliance needs (GDPR, PCI-DSS, HIPAA). Define security user stories: "As a user, my data must be encrypted at rest."
●
🎨 2. Design
Perform threat modeling (STRIDE). Define trust boundaries, authentication flows, and encryption strategies. Get security architecture reviewed before a line of code is written.
●
💻 3. Development
Apply secure coding standards. Use SAST tools (static analysis) in the IDE. Conduct peer code reviews with a security checklist. Validate all inputs, encode all outputs.
●
🧪 4. Testing
Run DAST (dynamic analysis), vulnerability scanning, and penetration testing. Test for OWASP Top 10. Fuzz inputs to find edge cases. Use automated security test suites in CI/CD.
●
🚀 5. Deployment
Harden server configurations. Enforce TLS, disable debug modes, set security headers. Use WAF (Web Application Firewall). Control secrets via vault, not environment files.
●
🔧 6. Maintenance
Apply patches promptly. Monitor logs for anomalies. Run incident response drills. Re-run threat modeling when adding major features. Track CVEs for all dependencies.
