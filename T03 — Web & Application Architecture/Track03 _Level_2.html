🪪

OpenID Connect (OIDC) — Who Are You?

OAuth answers "What can you do?" — OIDC answers "Who are you?"

OIDC sits on top of OAuth 2.0 and adds an ID Token — a signed JWT that contains the user's verified identity (name, email, profile picture).

OAuth 2.0

Issues an Access Token — grants permission to access resources. Doesn't tell you who the user is.

OIDC + OAuth 2.0

Issues an ID Token — verifies user identity. Now you know who they are AND what they can do.

💡 Real World: Google Sign-In uses OIDC. When you "Sign in with Google", it provides your Google account identity (name, email) via an ID Token, so the app knows who you are without ever handling your password.

Microservices & Service Mesh

Modern apps are built from small, independent pieces — and securing the communication between them is critical.

🧩

✅ Security Win: If the Notifications service is hacked, the Payment service is still safe. Fault isolation means a breach in one service doesn't automatically compromise everything.
🕸️

Service Mesh — Securing Service-to-Service

A service mesh handles all communication between microservices — adding encryption, auth, and observability automatically.

🔐 Mutual TLS (mTLS) — Both sides prove identity

In regular TLS, only the server has a certificate. With mTLS, both the client and server present certificates. This means Service A can be certain it's talking to Service B — not a rogue service injected by an attacker.

📊 Observability — See everything

Automatically collects metrics, logs, and traces from every service. If something goes wrong (or gets hacked), you can trace exactly which service was affected and when.

🚦 Traffic Control — Fine-grained rules

Define rules like "Service A can only call Service B, not Service C". This principle of least privilege limits blast radius if a service is compromised.
